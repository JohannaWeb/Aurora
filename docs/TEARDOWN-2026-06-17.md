# Aurora: A Brutal Teardown

*Written from the perspective of a Chromium engineer doing a design review. Dated 2026-06-17, against branch `feature/youtube-support-fix-no-rendering`. Every claim cites code.*

> TL;DR: Aurora has made the right first move: live pixels now come from Blitz/Stylo, and many JS mutations are pushed into the Blitz document directly instead of going through an HTML string. But the architecture is still two browsers in a coat. JavaScript mutates a hand-rolled DOM, rendering paints a Blitz DOM mirror, and the correctness of the whole thing depends on a large pile of bespoke synchronization glue, fake Shadow DOM, and YouTube-specific Polymer emulation.

---

## 0. The one diagram that explains the current branch

```
                JS (V8) runs YouTube's Polymer bundle
                          │  mutates
                          ▼
          ┌───────────────────────────────┐
          │  LEGACY hand-rolled DOM        │   src/dom/*
          │  (Rc<RefCell<Node>>)           │   JS sees THIS
          └───────────────┬───────────────┘
                          │ direct mutation sync when the bridge remembers
                          │ sync_append_child / sync_set_attribute / ...
                          ▼
          ┌───────────────────────────────┐
          │  BLITZ DOM + Stylo             │   src/blitz_document.rs
          │  (the thing that paints)       │   pixels come from HERE
          └───────────────┬───────────────┘
                          │ fallback when the mirror is missing or stale
                          ▼
                    try_from_dom() full snapshot rebuild

   Still alive in parallel:
   ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐
   │ legacy CSS   │  │ legacy style │  │ legacy LayoutTree    │
   │ src/css/*    │  │ src/style/*  │  │ src/layout/* (taffy) │
   └──────────────┘  └──────────────┘  └──────────────────────┘
```

The old "serialize the whole DOM and re-parse HTML on every reflow" critique is partly obsolete on this branch. `WindowInput::sync_blitz_snapshot` now calls `BlitzDocument::try_from_dom` (`src/window/input.rs:92-103`), and `BlitzDocument` has direct mutation hooks for append, insert, remove, replace, attributes, text, shadow roots, and child replacement (`src/blitz_document.rs:308-633`). That is meaningful progress.

But the system still has two independent document models. The one JavaScript sees is not the one Stylo owns. The renderer document is a mirror, and mirrors are where browser engine correctness goes to die.

---

## 1. The old fatal seam is now a fallback seam, but it is still a seam

The current live path builds Blitz from the legacy tree before scripts run (`src/runner/pipeline.rs:31-40`) and hands that Blitz document into V8 so DOM mutations can be applied directly. Good. That is the first architecture change that actually points toward a browser.

The problem is that the legacy DOM is still authoritative. When the snapshot is dirty, Aurora rebuilds the Blitz document from the entire legacy DOM (`src/window/input.rs:92-119`). `mark_dirty` still unconditionally sets both `needs_reflow` and `blitz_snapshot_dirty` (`src/window/input.rs:127-132`). The paint path still responds to a failed Blitz paint by marking the snapshot dirty and scheduling another reflow (`src/window/app.rs:199-206`).

So the new system has two modes:

- **Happy path:** every JS-visible mutation goes through a bridge hook and is mirrored into Blitz incrementally.
- **Recovery path:** rebuild the Blitz document wholesale from the legacy DOM.

That is better than stringifying the page every frame, but it is not the architecture Chromium would accept. A browser DOM cannot be "eventually mirrored" into the render tree by dozens of side-effect calls sprinkled through bindings. Every missing hook creates a split-brain page: JS sees one tree, Stylo paints another, hit testing maps through stale IDs, and layout-dependent JS reads become roulette.

Severity: still architectural. The O(n) HTML round-trip is mostly gone from the primary path; the two-DOM contract is not.

---

## 2. The mirror is hand-maintained, and the bug class is obvious

`BlitzDocument` maintains two maps, `legacy_to_blitz` and `blitz_to_legacy` (`src/blitz_document.rs:132-138`). Every mutation helper must update both the Blitz tree and those maps. You can see the surface area:

- `sync_append_child` creates Blitz IDs for the incoming legacy child and appends them (`src/blitz_document.rs:308-333`).
- `sync_insert_before` does the same, but separately handles the anchor (`src/blitz_document.rs:336-371`).
- `sync_remove_child` removes the Blitz node and then manually removes subtree mappings (`src/blitz_document.rs:374-388`).
- `sync_replace_child` creates the new subtree, swaps it into Blitz, and removes the old subtree mapping (`src/blitz_document.rs:391-429`).
- `sync_replace_children` recreates children from the legacy DOM and appends them (`src/blitz_document.rs:600-633`).

This is not incremental DOM in the browser-engine sense. This is a custom replication protocol. The protocol is not declarative, transactional, or centrally validated. It is a set of imperative calls hidden behind V8 binding code. Miss one, call one in the wrong order, throw during one, or mutate a structure whose Blitz representation is synthetic rather than native, and the document diverges.

The code already betrays that fear. `WindowInput` keeps a dirty snapshot bit (`src/window/input.rs:29`) and rebuilds from scratch if it thinks the mirror might need it (`src/window/input.rs:92-119`). That fallback exists because the bridge cannot prove it is complete.

---

## 3. Shadow DOM is still not Shadow DOM

This branch no longer merely dumps shadow children into the host's light child list. `attach_shadow` stores a distinct `shadow_root` document fragment on the element (`src/js_v8/node_create.rs:2263-2275`). That is another real improvement.

But the Blitz representation is still a synthetic light-DOM node:

```rust
mutator.create_element(element_qual_name_from_str("div", &host_ns), attrs)
```

with `data-aurora-shadow-root`, `data-aurora-shadow-mode`, and `data-aurora-shadow-host` attributes (`src/blitz_document.rs:525-570`). In other words: Stylo is not styling a native shadow tree. It is styling a `<div>` child with marker attributes.

That means platform semantics are still approximated:

- `:host`, `:host-context`, `::slotted`, slot distribution, composed tree traversal, focus delegation, and event retargeting are not native rendering behavior.
- Selector visibility is patched manually in query helpers by checking nearest synthetic shadow roots (`src/blitz_document.rs:899-915`).
- CSS scoping is rewritten in JavaScript by `ShadyCSS-lite` (`src/js_polyfills/custom_elements.js:354-470`).

This is the right direction only if it is a stepping stone. As a destination, it is a trap. Modern YouTube is Polymer/Web Components machinery layered over thousands of scoped styles. Faking the tree shape with marker attributes means every downstream feature has to ask, "is this a real DOM rule, or Aurora's synthetic shadow convention?"

---

## 4. ShadyCSS-lite is a symptom, not a solution

`custom_elements.js` says the quiet part out loud. Its comment still describes a flattened/serialized world (`src/js_polyfills/custom_elements.js:355-365`), even though the current branch now has direct Blitz mutation and template serialization support. The comment is stale, but the architecture smell remains: the project is compensating for missing platform behavior in a giant JS shim.

The selector rewriter:

- scopes component-internal selectors as `tag selector` (`src/js_polyfills/custom_elements.js:405-406`);
- rewrites `:host(x)` by textual replacement (`src/js_polyfills/custom_elements.js:397-400`);
- rewrites `::slotted(x)` to `tag x` (`src/js_polyfills/custom_elements.js:401-403`);
- walks CSS blocks with a hand parser that counts braces and only recurses into `@media` and `@supports` (`src/js_polyfills/custom_elements.js:415-449`).

This is not CSS scoping. It is a regex-and-brace-counter approximation of a browser feature. It will over-match nested same-tag components, under-match real slot distribution, and fall apart on enough modern CSS edge cases to keep producing "unstyled but stamped" pages.

The good news: template serialization is no longer the smoking gun it was. `serialize_html.rs` now serializes `template_contents` (`src/dom/serialize_html.rs:35-42`), and `innerHTML` correctly targets a template's content fragment (`src/js_v8/node_create.rs:1676-1688`). The bad news: the active YouTube problem was never just "a style tag did not serialize." It is that the renderer does not have native Shadow DOM styling semantics.

---

## 5. The legacy style/layout engine is still undead

The live renderer uses Blitz (`src/window/input.rs:26-27`). But Aurora still carries the old CSS, style, and layout stacks. In normal Blitz mode, `WindowInput::reflow` skips legacy layout unless there is no Blitz document or `AURORA_LEGACY_LAYOUT_SYNC` is enabled (`src/window/input.rs:56-64`). Startup does the same, using a placeholder layout when Blitz exists and debug flags are off (`src/runner/pipeline.rs:46-68`).

This is progress, but it creates an awkward split:

- Pixels come from Blitz.
- JS layout accessors and some test/debug paths still depend on legacy layout state, or on a placeholder when it is disabled.
- Image, SVG, and media discovery in the legacy path only happen when that old layout path runs (`src/window/input.rs:61-83`).

Chromium has many trees, but they are layered products of one DOM and one style system. Aurora has a painted Blitz tree and a legacy layout tree that can be absent, placeholder-backed, or env-var-resurrected. That is not a layering strategy; it is a compatibility pact with old code.

Delete the old layout path as an authority. If a JS layout accessor is needed, make it read from the Blitz/Stylo layout data that actually produced the pixels.

---

## 6. Panic handling improved from "instant white page" to "retry loop"

The old code permanently dropped `blitz_doc` after a paint failure. The current paint path is less catastrophic: a failed `paint_to_scene` marks the Blitz snapshot dirty and schedules another reflow (`src/window/app.rs:199-206`). `BlitzDocument` also allows up to `MAX_CONSECUTIVE_PANICS = 5` before marking itself unhealthy (`src/blitz_document.rs:54`, `src/blitz_document.rs:643-662`).

That is better operational behavior. The old global panic-hook swap is gone too; `catch_stylo_panic` is now just `panic::catch_unwind` plus logging (`src/blitz_document.rs:1030-1041`). Good. That is the right shape for isolating Stylo failures.

The remaining user-visible failure mode is odd: while healthy, `paint_to_scene` can report success after a failed paint because it returns `painted || self.healthy` (`src/blitz_document.rs:661`). That avoids a white page, but it can also hide the fact that the scene for this frame was not actually painted. A browser should preserve the last known-good frame deliberately, mark the renderer degraded observably, and recover on the next successful style/layout pass.

---

## 7. The event loop is still not the web event loop

`pump_ready_work` has improved from the old "teleport virtual time to quiescence" design. It now sleeps in small chunks toward timer/rAF deadlines (`src/runner/pipeline.rs:267-278`) and drains animation frames against a 16 ms frame interval (`src/runner/pipeline.rs:247-252`).

But this is still a boot-time pump around a captured page load, not a browser event loop. Mutation records, timers, animation frames, script execution, layout, paint, network, media, input, and microtasks are not integrated into the same event-loop model that a large SPA expects. The code has a dedicated post-pass to "sweep" Polymer bindings that were stamped outside Aurora's upgrade pipeline (`src/runner/pipeline.rs:285-304`). That is an admission that framework execution order is not naturally falling out of platform semantics.

YouTube boot is timing-sensitive because it is lazy, scheduler-heavy, and observer-heavy. A browser does not make it work by draining enough callbacks. It makes it work by implementing the event loop, task queues, microtask checkpoints, rendering opportunities, observers, and frame lifecycle in the same semantic order the platform specifies.

---

## 8. So, specifically, why is YouTube still fragile or blank?

Most likely failure stack, updated for the current branch:

1. **The rendered tree is a mirror, not the JS tree.** Direct mutation sync is a big improvement, but correctness depends on every DOM operation, parser insertion, template stamp, text update, attribute change, and shadow mutation hitting the right `sync_*` method in the right order. Any missed edge produces a page that exists to JS but not to Stylo.
2. **Shadow DOM styling is synthetic.** The Blitz tree represents shadow roots as marker `<div>` elements (`src/blitz_document.rs:525-570`), then JS rewrites scoped CSS (`src/js_polyfills/custom_elements.js:354-470`). YouTube's layout depends heavily on scoped component CSS, custom properties, and slot-like composition. Approximate scoping yields approximate layout.
3. **The framework is still being impersonated.** The codebase contains a huge YouTube/Polymer emulation layer with binding sweeps and component-specific rescue paths. The more of YouTube Aurora implements in JS shims, the more every YouTube bundle update becomes a browser-engine regression.
4. **Failure recovery can keep repainting stale or empty scenes.** Paint failures dirty the Blitz snapshot and schedule reflow (`src/window/app.rs:199-206`), while `paint_to_scene` can return healthy even when the current paint failed (`src/blitz_document.rs:643-662`). That is better than killing the renderer, but it can make "nothing changed on screen" harder to diagnose.

The blank page is probably no longer caused by a single missing serializer walk. It is caused by the bigger fact that YouTube is a modern component app and Aurora still does not give it a native component platform.

## 9. What a Chromium engineer would actually tell you to do

Ranked by leverage, not by effort:

1. **Finish the migration to one authoritative DOM.** The winner should be the DOM that Stylo/Blitz can style and paint incrementally. JS bindings should mutate that DOM directly, not a legacy tree that must be mirrored into it.
2. **Make Shadow DOM real in the renderer.** A synthetic `<div data-aurora-shadow-root>` is useful scaffolding, but it cannot be the final model. Native shadow-tree style scoping, slot distribution, composed-tree traversal, and event retargeting are table stakes for YouTube.
3. **Retire ShadyCSS-lite as soon as native shadow styling exists.** Regex CSS rewriting is not a platform implementation. It is a debugging bridge.
4. **Delete legacy layout as an authority.** JS layout accessors, hit testing, screenshots, and media discovery should read from the same tree that produces pixels.
5. **Make the event loop a first-class subsystem.** Stop relying on boot-time sweeps and callback draining as correctness tools. Implement task/microtask/rendering-order semantics explicitly.
6. **Make paint failure state observable.** The hook swap is gone, which is good. Now preserve last good frames intentionally, surface repeated Stylo failures, and make recovery state visible in logs/debug UI.
7. **Remove YouTube-specific fallbacks once the platform works.** Component-specific rescue code is fine as temporary instrumentation. It is poisonous as a compatibility strategy.

---

## 10. The honest summary

Aurora has clearly moved in the right direction since the first version of this teardown. The direct Blitz mutation hooks are real progress. Template content serialization and template-aware `innerHTML` are real progress. The renderer no longer appears to be permanently discarded on the first Stylo panic. Good.

But the project is still paying interest on the original architecture. A modern site like YouTube does not need a better impersonation of Polymer; it needs the platform Polymer assumes exists. As long as JavaScript runs against a hand-rolled DOM while Stylo paints a mirrored Blitz DOM with synthetic shadow roots, every fix will feel like tightening one screw and hearing another panel buzz.

Collapse the architecture to one DOM. Make Shadow DOM and the event loop boringly correct. Then the YouTube-specific code can shrink instead of growing.
