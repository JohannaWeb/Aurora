# Aurora: A Brutal Teardown

*Written from the perspective of a Chromium engineer doing a design review. Dated 2026-06-17, against branch `feature/youtube-support-fix-no-rendering`. Every claim cites code.*

> TL;DR: Aurora is not one browser engine. It is **four half-engines bolted together with a string serializer**, and the seam between them is re-walked, re-parsed, and re-styled from scratch on every mutation. YouTube does not render because the path that produces pixels (Blitz/Stylo) and the path that runs the page (the hand-rolled DOM + a 2,200-line JS Polymer emulator) only communicate by **printing the DOM to an HTML string and re-parsing it**. That is not a rendering bug. That is the architecture working as designed.

---

## 0. The one diagram that explains everything

```
                JS (V8) runs YouTube's Polymer bundle
                          │  mutates
                          ▼
          ┌───────────────────────────────┐
          │  LEGACY hand-rolled DOM        │   src/dom/*
          │  (Rc<RefCell<Node>>)           │   ← JS sees THIS
          └───────────────┬───────────────┘
                          │ serialize_outer_html()  ← prints to a String
                          ▼
                   "<!DOCTYPE html>..."          src/dom/serialize_html.rs
                          │ try_from_html()  ← re-parses from scratch
                          ▼
          ┌───────────────────────────────┐
          │  BLITZ DOM + Stylo             │   src/blitz_document.rs
          │  (the thing that paints)       │   ← PIXELS come from HERE
          └───────────────────────────────┘

   Also alive, in parallel, never painted:
   ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐
   │ legacy CSS   │  │ legacy style │  │ legacy LayoutTree    │
   │ src/css/*    │  │ src/style/*  │  │ src/layout/* (taffy) │
   └──────────────┘  └──────────────┘  └──────────────────────┘
```

There are **two complete, independent style+layout systems** in this codebase. One of them (the hand-rolled `src/css` + `src/style` + `src/layout`, ~6k LOC) computes layout that is *thrown away* — it exists only for "tests, screenshots, JS layout accessors, and hit testing" (`src/window/app.rs:62-65`). The other (Stylo, via Blitz) is the only thing that produces visible output. They are reconciled by **re-parsing an HTML string**.

No shipping browser has ever worked this way, and there is a reason.

---

## 1. The cardinal sin: serialize → re-parse on every reflow

`src/window/input.rs:71-86`:

```rust
if self.blitz_doc.is_some() {
    // Re-serialize the mutated legacy DOM to HTML, then reload it into blitz_doc.
    let html = crate::dom::serialize_outer_html(&self.dom);
    self.blitz_doc = BlitzDocument::try_from_html(&html, ...);
}
```

This runs on **every** `needs_reflow` (`src/window/app.rs:44-58`), which is set on every JS mutation batch (`mark_dirty`, `take_needs_reflow`). For a page like YouTube that mutates continuously during boot, this means:

- **Full HTML re-serialization of the entire document, every frame.** O(nodes) string building, 4 KB initial buffer (`serialize_html.rs:4`) for a document that is megabytes.
- **Full re-parse and full re-style of the entire document, every frame.** Blitz/Stylo throws away *all* incremental state — the style cache, the layout tree, the box tree — and rebuilds from a cold string. Chromium spent 15 years building style invalidation, layout-tree reuse, and `LayoutNG` caching specifically so it would **never** do this. Aurora does it unconditionally.
- **Everything not expressible in serialized HTML is destroyed every frame:** Blitz-side scroll position, focus, input selection/caret, form control state, running CSS animations/transitions, `<canvas>` contents, constructed/adopted stylesheets, and any DOM identity. The Blitz document has no memory; it is reincarnated each tick.

This is not an optimization gap. It is an O(n) full-document rebuild on the hot path, and it caps the achievable frame rate of *any* dynamic page at "however long it takes to re-parse the whole DOM." The commit message on the tip of this branch — *"rendering should not be a fixed magic number hate this"* — is the right instinct pointed at the wrong line. The magic number isn't the problem. The reincarnation is.

**Severity: fatal.** This single design choice makes a smooth SPA impossible regardless of how good the JS shims get.

## 2. The serializer is lossy, and it loses *exactly* what YouTube needs

`src/dom/serialize_html.rs` is 80 lines. A browser's serializer is the easy part; a browser's *parser round-trip fidelity* is the hard part, and this one isn't trying. Concretely:

- **`<template>` content is invisible.** Polymer component styles live in `<dom-module><template>`, i.e. `Node.template_contents`, which the serializer never walks. So a component's `<style>` simply does not exist in the string handed to Stylo. This is *the* YouTube CSS gap, and the "fix" is a JS shim (§4) that re-implements CSS scoping by hand. The serializer being lossy here is the root cause that everything downstream is compensating for.
- **No namespaces, no foreign content.** SVG/MathML attributes, `xmlns`, case-sensitive foreign elements — all flattened to naive `<tag attr="v">`.
- **Attribute escaping is hand-rolled** (`html_escape`, line 74) and order-dependent; raw-text detection is a hardcoded match on `"script" | "style"` (line 30). Anything Stylo's real HTML parser treats specially that this serializer doesn't will silently diverge.
- **The string is the only API between the two DOMs.** Any DOM state that has no HTML attribute representation — IDL properties set from JS, `el.value`, `el.checked`, computed `el.style` that the legacy side tracks out-of-band — may or may not survive the round trip depending on whether some code path mirrored it into an attribute.

A serializer is supposed to be a debugging/`outerHTML` convenience. Here it is load-bearing structural infrastructure, and it drops the payload.

## 3. Shadow DOM is faked by flattening it away — then re-faked in JS

`attach_shadow` (`src/js_v8/node_create.rs`) returns a proxy that **reuses the host's `node_id`** and appends "shadow" children into the host's light children. There is no shadow tree. This has two consequences a Chromium engineer will flag instantly:

1. **Encapsulation is gone.** `:host`, `::slotted`, `:host-context`, slot distribution, and scoped selector matching match nothing, because there is nothing scoped — it's all one flat light tree.
2. **So CSS scoping is re-implemented in JavaScript, by string-rewriting selectors.** `src/js_polyfills/custom_elements.js:354-476` ("ShadyCSS-lite") hoists each component's `<style>` into `<head>` and rewrites `:host(x)` → `tagx`, `::slotted(x)` → `tag x`, and scopes bare rules as `tag <sel>`. This is a hand-rolled, regex-driven re-implementation of a problem the platform solved with ShadyCSS — except the original ShadyCSS had per-instance scope classes. This one **scopes by tag name only** (custom_elements.js comments admit it), so two nested instances of the same component over-match each other's styles. And critically:

   > **It only hooks the Polymer 2 `<dom-module>` path, not Polymer 3 `static get template`.**

   Modern YouTube (the "kevlar" desktop bundle) is overwhelmingly Polymer 3. Those components define their template/styles via a static getter, never register a `<dom-module>`, and therefore **never get their `<style>` hoisted**. Result: stamped-but-unstyled components, collapsed to ~0 height. (This is also why the debug dumper in `blitz_document.rs` exists — to hunt "where YouTube's height collapses to ~0," per the comment at line 124-125.)

This is the proximate cause of the blank page. See §7.

## 4. The JS layer is a 2,200-line emulator fighting the framework

`src/js_polyfills/custom_elements.js` is 2,226 lines. It is not a polyfill; it is a partial re-implementation of:

- the **custom-elements upgrade machinery** (an `upgradeStack`, a patched `HTMLElement` constructor doing `Reflect.construct` gymnastics, lines 33-81),
- **Polymer's property-effects and data-binding system** (`[[prop]]`/`{{prop}}` parsing, computed bindings, `notifyPath`, `_propertiesChanged` wrapping — lines 1646+),
- **`dom-repeat` / `dom-if`** from scratch (`polymer_shim.js:219-318`),
- **Polymer's `$` id-map** (`rebuildPolymerIdMap`, line 826),
- and a pile of **per-component special cases**: `yt-attributed-string` prop sanitizing, a `ytd-rich-grid-renderer` fallback that *synthesizes placeholder boxes* (`installRichGridFallback`, line 1148+), `ytd-feed-nudge-renderer` fallbacks, `ytd-rich-section-renderer` fallbacks...

And `src/runner/pipeline.rs:211` (`drive_polymer_page_manager_navigation`) reaches into YouTube's closure-private navigation state machine, gives up on it, and calls `pageManager.updatePageData(getInitialData())` **directly** because "the initial transition never fires in our environment."

This is the signature of a browser that doesn't actually run the framework — it *impersonates the parts of the framework it has reverse-engineered*, one component at a time. Every YouTube A/B test, every bundle update, every new renderer tag is a new special case. This is unwinnable as a maintenance strategy. Chromium runs Polymer because it implements the *platform* (custom elements, shadow DOM, the event loop) correctly; YouTube's framework then "just works." Aurora implements YouTube, not the platform.

The presence of `installRichGridFallback` *synthesizing grey "aurora rich-section" debug boxes into `document.body`* (lines 1148-1644) is the tell: the real render path doesn't produce content, so a fallback paints fake content to prove something reached the screen.

## 5. The event loop is a fiction

`pump_ready_work` (`src/runner/pipeline.rs:269-320`) runs timers and `requestAnimationFrame` against a **virtual clock that jumps forward to the next scheduled callback**, bounded by a 2 s virtual budget and a 5 s wall-clock budget. There is no real event loop, no real rAF cadence, no real microtask/task ordering tied to wall time.

For a page that hydrates "through chained `setTimeout`s" (the comment's own words), this means YouTube's boot is being *simulated to quiescence in a burst*, then frozen. Anything that depends on real time — IntersectionObserver tied to scroll, debounced layout, lazy hydration, media timeupdate cadence — diverges from real-browser semantics. You cannot get video playback timing right against a clock that teleports.

## 6. Failure handling turns one Stylo panic into a permanent white page

`src/blitz_document.rs:109-122, 197-206`: paint and resolve are wrapped in `catch_stylo_panic`, and on *any* panic the document sets `healthy = false`. Then `src/window/app.rs:188-192`:

```rust
if !blitz_doc.paint_to_scene(...) {
    app.input.blitz_doc = None;   // renderer gone for the rest of the session
    content_scene = Scene::new(); // blank
}
```

So a **single** Stylo panic — on a page whose CSS is megabytes of modern features Stylo may not love — permanently disables rendering for the whole session and yields a white page with no recovery. Given YouTube's CSS surface area, this is a live grenade.

Worse, `catch_stylo_panic` (lines 209-228) **swaps the process-global panic hook** under a mutex on every paint and resolve. That is a global side effect on the hot path, racing every other thread that might panic, and serialized behind a single global lock. Two windows / background fetch threads / the JS engine panicking concurrently will interleave with this hook-swap in ways that are, charitably, undefined.

## 7. So, specifically, why is YouTube blank?

Chaining the above, the most likely failure stack (in order of confidence):

1. **Component styles never reach Stylo.** Polymer 3 components (most of kevlar) define styles in `static get template`, which §3's ShadyCSS-lite shim doesn't hook, so their `<style>` is never hoisted into `<head>`, so it's never in the serialized HTML, so Stylo never sees it. Unstyled flex/grid containers collapse to 0 height → nothing visible even though the DOM is "there." This matches the `blitz_document.rs` debug tooling explicitly hunting "height collapses to ~0."
2. **Even the styles that *do* get hoisted are mis-scoped.** Tag-name-only scoping (§3) over-matches nested same-tag components and the regex rewriter doesn't robustly handle CSS custom properties (`--ytd-*`, which YouTube leans on heavily), `@media`/`@supports` nesting, or `::slotted` chains. Partial/incorrect CSS → broken layout.
3. **The serialize→reparse path may be dropping dynamically-stamped subtrees** that exist as `<template>` content or as JS-only state at the moment of serialization (§1, §2).
4. **A Stylo panic on YouTube's CSS would zero the page permanently** (§6), converting "partially broken" into "fully blank."

You don't need a debugger to predict this; the architecture guarantees at least #1 and #2.

## 8. What a Chromium engineer would actually tell you to do

Ranked by leverage, not by effort:

1. **Kill the serialize→reparse seam.** Pick *one* DOM. The only DOM worth keeping is the one Stylo/Blitz already owns. Drive YouTube's JS directly against a Blitz-backed DOM (real `blitz-dom` mutation API) so mutations are *incremental* and pixels reflect state without a round trip. This deletes §1, most of §2, and the entire raison d'être of the legacy `src/css`/`src/style`/`src/layout` stack (~6k LOC). This is the migration the memory calls "deferred until YouTube stabilizes" — that ordering is backwards. **YouTube cannot stabilize *until* this is done.**
2. **Implement real Shadow DOM in the chosen DOM**, or at minimum a correct ShadyCSS with per-instance scope classes for *both* Polymer 2 and Polymer 3 template paths. Stop string-rewriting selectors in JS regex.
3. **Run a real event loop** against wall time (or a deterministic virtual clock that you *advance in lockstep with frame production*, not one that teleports to quiescence and freezes).
4. **Stop disabling the renderer on panic.** Catch, log, recover, repaint next frame. Never `blitz_doc = None` permanently. And get the global panic-hook swap off the per-frame path.
5. **Delete the per-component fallback shims** (`installRichGridFallback` et al.) once real rendering works. They are debugging scaffolding masquerading as features, and they actively add misleading boxes (the code itself restricts them to debug runs for exactly this reason — they were lying about success).
6. Only *then* worry about the triple JS-engine duplication (`js_v8` / `js_sm` / `js_boa`, ~3 parallel copies of every DOM binding — `node_create.rs` is 2,242 lines in V8, `document/api.rs` is 4,253 in SM). It's debt, but it's not why YouTube is blank.

---

## 9. The honest summary

Aurora's team has done genuinely impressive reverse-engineering — driving Polymer's private boot, hand-rolling data binding, faking custom-element upgrades well enough that YouTube's bundle runs at all. That work is real.

But it's all being poured into the wrong shape. The project is trying to make YouTube render *despite* an architecture whose central data path — print the DOM to a string and re-parse it every frame — is fundamentally incompatible with a dynamic web app. No amount of JS shimming fixes a render pipeline that has amnesia. The 2,200-line emulator exists to compensate for the flattened shadow DOM; the flattened shadow DOM exists to compensate for there being two DOMs; the two DOMs exist because the painting engine was bolted on via a serializer instead of adopted as *the* DOM.

Collapse it to one DOM. Everything else on the list gets easier, and several thousand lines of compensating machinery simply evaporate.
