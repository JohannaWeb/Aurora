# Aurora: Brutal Teardown, 2026-06-18

*Written from the perspective of a Chromium engineer doing a design review. Dated
2026-06-18, against branch `feature/youtube-support-fix-no-rendering`. Every claim
cites code or your own docs. A prior teardown (`TEARDOWN-2026-06-17.md`) covered the
two-DOM mirror in depth; this one does not repeat it — it goes after the facts that
doc didn't say out loud.*

---

## The one number that matters

You ran 9 phases and 14 execution-order tasks. You wrote `MirrorIntegrityError`, a
transactional mutation bridge, a `ShadowTreeBackend` abstraction, canonical
event-loop phase ordering, native layout-metric bridges, and a full workaround
inventory. Real, disciplined engineering.

The result, per your own notes (`YOUTUBE_RENDERING_STABILIZATION_ACTION_PLAN.md:30`):
**a hamburger icon and three circles.** "Only the masthead chrome paints; no search
box, logo, guide, or feed."

And then the kill shot, in your own words (`:31`): the logged-out home feed contains
"**zero** video items," the real content "requires either continuation networking or
targeting a content-rich page... This is beyond the stabilization plan's current
phases."

Translate that: you spent the entire plan hardening the pipeline that *delivers*
content, and the content was never going to arrive, because **you don't make the
network calls that fetch it.** A Chromium engineer reads that and asks one question:
why was *any* of the Shadow-DOM / ShadyCSS / Polymer-mirror work prioritized before
you proved the page even has data to render? You optimized the projector before
checking there was film in the camera.

## The docs describe a browser that doesn't exist

`docs/ARCHITECTURE.md` is your arc42 system-of-record. It is fiction:

- `:39` "JavaScript must be handled by... **SpiderMonkey**." You run **V8**
  (`src/js_v8/`, `Cargo.toml`). There is no `js_sm`, no `js_boa`. The diagram at
  `:102` shows modules that don't exist.
- `:38` "**No dependency on Chromium, WebKit, Gecko, or Servo.**"

That last one is the big lie. Be precise about what Aurora actually is:

- Your JS engine is **V8** — Chromium's engine, downloaded as a **prebuilt binary
  blob at build time** (your own Cargo comment admits it).
- Your DOM, CSS, and layout come from **`blitz-dom`**, which is **Stylo** —
  *Firefox/Servo's* style engine.
- Your painter is **Vello / anyrender** — Linebender's.

So the "sovereign, no-Chromium, no-Servo, Rust browser engine" is **Chromium's JS
engine driving Servo's CSS engine, painted by someone else's GPU rasterizer, glued
together in Rust.** Sovereignty Goal #5 in your own quality table is marketing copy
contradicted by line 39 of your own Cargo.toml. The thing that is genuinely *yours*
is the ~25k lines of glue and shims holding three foreign engines in a trenchcoat —
and that glue is the bug farm.

## Two DOMs, hand-mirrored — yes, still

The 06-17 teardown nailed this and it's still true: JS mutates a hand-rolled
`Rc<RefCell<Node>>` tree (`src/dom/`), and Stylo paints a *different* tree
(`BlitzDocument`), kept in sync by a hand-written replication protocol
(`legacy_to_blitz` / `blitz_to_legacy` maps + a dozen `sync_*` methods). You even
built `validate_mirror_integrity` and `MirrorIntegrityError` — which is the tell.
**You don't write an integrity validator for an invariant you trust.** That class is
a confession that the two trees drift, and you're now paying to *detect* the drift
instead of eliminating its possibility. Chromium has one tree and derived trees
computed from it. You have two authored trees and a referee.

And you're *still* eating it in production: `:30` notes "5 caught
`blitz-dom mutator.rs:807 'unreachable'` panics during attribute replacement that
each force a snapshot rebuild." That's the mirror falling over five times on a single
page load and getting rebuilt from scratch.

## 4,027 lines of JavaScript impersonating a platform

```
custom_elements.js   1801
v8_post.js            852
v8_base.js            784
polymer_shim.js       444
```

`custom_elements.js` alone is a from-scratch reimplementation of custom-element
upgrades, ShadyDOM event delivery, and **CSS scoping done by counting braces in a
regex** (`scopeCss` / `rewriteScopedSelector`). Your own workaround inventory lists
~13 rescue paths, each with a "delete condition" that amounts to "when the engine
actually does its job." This isn't a browser running YouTube; it's a YouTube-shaped
emulator. Every Polymer bundle YouTube ships is now a potential **engine regression
for you**, and you have no test against real YouTube — only fixtures you authored to
match the shims you wrote. That's a closed loop that proves nothing.

## Running hostile code with a hair trigger

- **294** `unwrap()` / `expect()` / `panic!` / `unreachable!` across `src/`.
- **115** `unsafe` blocks.

This engine's stated purpose is to load **youtube.com** — arbitrary, adversarial,
third-party JavaScript and CSS — and the codebase is studded with ~300 ways to abort
the process on unexpected input. You've wrapped Stylo in `catch_unwind` and added
`MAX_CONSECUTIVE_PANICS = 5`, which is the right instinct, but it's a tourniquet on a
design that panics as a control-flow strategy. A browser treats untrusted input as
the *normal* case. Yours treats it as a series of assertions waiting to be violated.

~30 branches. Including `javascript/move-to-spider-monkey-rust-bindings` —

## What's actually good (so the brutality is honest)

- Moving live pixels onto Blitz/Stylo instead of re-parsing HTML strings every reflow
  was the correct call and it's done.
- `PaintResult` + last-known-good frame preservation is real operational maturity most
  hobby engines never reach.
- Isolating synthetic shadow behind `ShadowTreeBackend` and gating diagnostics behind
  env flags is clean, disciplined work.
- The workaround inventory with explicit delete-conditions is more architectural
  honesty than most *shipping* engines maintain.

You are a genuinely good engineer building on a strategically doomed foundation, and
your own docs are the only thing lying about it.

## What a Chromium engineer tells you to do, ranked by leverage

1. **Fix the goal, not the pipeline.** You proved the logged-out home page has no
   content to render. Stop polishing the renderer. Either (a) make
   continuation/network fetches so a real feed exists, or (b) retarget to a
   watch/search page whose initial data actually carries content. Until there's data,
   every rendering improvement is invisible by construction.
2. **Collapse to one DOM.** Delete the legacy `Rc<RefCell>` tree as an authority. JS
   bindings mutate the Blitz/Stylo tree directly. The day you do this,
   `MirrorIntegrityError`, the sync protocol, and the snapshot-rebuild fallback all
   get deleted, not hardened.
3. **Rewrite ARCHITECTURE.md today.** It actively lies about your JS engine and your
   dependencies. A stale system-of-record is worse than none — it makes new
   contributors (and future-you) reason about a browser that was deleted months ago.
4. **Drop the "sovereignty / no-Servo / no-Chromium" framing** or define it honestly.
   You depend on V8 *and* Stylo. Own that — "Rust-native integration of best-in-class
   engines" is a defensible pitch. "No Chromium dependency" while statically linking
   Chromium's JS engine is not.
5. **Decriminalize bad input.** Burn down the 294 panics on any path that touches page
   content. Hostile input is the spec, not the exception.
6. **Delete the impersonation as the platform improves.** Your inventory already says
   when each shim dies. Hold that line — every shim that becomes load-bearing is a
   permanent tax paid to Google's frontend team.

## The honest summary

Aurora isn't failing on craft — the craft is good. It's failing when the thesis is
too broad. Treating YouTube as a hostile modern-web benchmark is defensible only if
the first gate stays narrow: bootstrap enough application data, DOM mutation,
custom elements, style, and layout to render one real content-bearing route
reliably. Chasing full YouTube rendering or playback first means picking the single
most hostile, framework-saturated, network-coupled target on the web — and after
all the work, the page is a hamburger and three circles because the data never
arrives. Pick a target you can actually finish, make the architecture one DOM, and
make the docs tell the truth about the benchmark.
